# Prompt Studio リグレッションテスト仕様 v0.3

## 1. 概要

リグレッションテスト機能は、**プロンプト（Prefix）を一度だけ LLM に prefill し、その時点の KV キャッシュを保存**したうえで、複数のテスト入力（Suffix）を高速に実行・判定する機能である。

目的は以下とする。

* プロンプト変更後の人格・口調・制約の退行を検出する
* 長大な Prefix の prefill コストをテスト実行ごとに繰り返さない
* テストケースを Git 管理可能な YAML で定義する
* 既存の RUN TEST フロー（編集 → ビルド → 実行）を拡張する

---

## 2. 位置づけと責務

### 2.1 v0.1 仕様との関係

v0.1 仕様では「KVキャッシュ生成」「推論高速化」は **配信ランタイムの責務外** としている。

本機能はこれと矛盾しない。Prompt Studio が行うのは次のみである。

| 責務 | 担当 |
|------|------|
| KV の生成・保持・復元 | llama-server（llama.cpp） |
| スロット save/restore の orchestration | Prompt Studio Backend |
| テストケース定義・結果表示 | Prompt Studio（Frontend + Backend） |
| 配信時の KV 管理 | 配信ランタイム（本機能の対象外） |

Prompt Studio は **開発時の回帰検証** に KV を利用する。配信用キャッシュ基盤ではない。

### 2.2 現状（v0.2）との差分

| 項目 | v0.2 RUN TEST | v0.3 Regression |
|------|---------------|-----------------|
| 入力 | 完成 Prompt 全文を 1 リクエスト | Prefix（完成 Prompt）+ テスト入力 |
| 実行 | 単発 | スイート単位で複数ケース |
| 判定 | 目視 | ルールベース自動判定 + 目視 |
| 速度 | Prefix を毎回 prefill | Prefix KV を再利用 |

## 3. 用語

| 用語 | 説明 |
|------|------|
| **Prefix** | 完成 Prompt 全文。Components から `build.yaml` 順に結合した静的部 |
| **Suffix / Test Input** | 各テストケースのユーザー入力（例: 「こんにちは」） |
| **Snapshot** | ある Prefix 状態に対応する KV キャッシュ + メタデータのセット |
| **Suite** | 複数テストケースの集合（YAML 定義） |
| **Run** | Suite を 1 回実行した結果（レポート） |
| **Slot** | llama-server が管理する KV キャッシュ単位（`id_slot`） |

---

## 4. 処理フロー

### 4.1 全体像

```text
[編集・自動ビルド]
        ↓
[Snapshot 更新] ── Prefix を prefill → slot save → メタ保存
        ↓
[Regression Run] ── slot restore → Suffix ごとに推論 → 判定
        ↓
[結果表示] ── Pass/Fail・差分・メトリクス
```

### 4.2 Snapshot 作成フロー

```text
1. 現在の build.yaml + Components から Prefix を生成
2. Prefix ハッシュ（prompt_hash）を計算
3. llama-server slot 0 を erase（任意・推奨）
4. Prefix を system メッセージとして slot 0 に prefill
   （chat/completions, id_slot=0, cache_prompt=true）
5. POST /slots/0?action=save → workspace/regression/kv/{prompt_hash}.bin
6. meta.yaml を書き込み
```

### 4.3 テスト実行フロー（1 ケース）

```text
1. Snapshot の prompt_hash と現在 Prefix の hash を比較
   → 不一致なら WARN（実行は可能だが結果の信頼性低下）
2. POST /slots/0?action=restore （filename: {prompt_hash}.bin）
3. chat/completions:
   - messages: [system: Prefix, user: Test Input]
   - id_slot: 0
   - cache_prompt: true
4. 応答テキストを matchers で判定
5. usage / timings を記録
```

### 4.4 高速化の原理

Prefix トークン数が N、テスト入力が M、ケース数が K のとき:

| 方式 | Prefix prefill 回数 | 概算 |
|------|---------------------|------|
| 従来（RUN TEST × K） | K 回 | O(K × N) |
| Regression（Snapshot 利用） | 1 回（Snapshot 更新時のみ） | O(N + K × M) |

M ≪ N かつ K が大きいほど効果が大きい。

## 5. llama-server 要件

### 5.1 必須設定

`config/llama.yaml` に以下を追加する（`extra_args` 経由）。

```yaml
extra_args:
  - --slot-save-path
  - workspace/regression/kv
  - -sps
  - "0.0"          # 自動スロット選択を無効化（明示 slot 利用）
```

| オプション | 目的 |
|------------|------|
| `--slot-save-path` | `/slots/{id}?action=save\|restore` を有効化 |
| `-sps 0.0` | 回帰テスト専用 slot を固定利用 |
| `--parallel 1` | v0.3 初期は単一 slot 前提（後述の将来拡張で変更可） |

### 5.2 API 利用

| 操作 | エンドポイント |
|------|----------------|
| 状態確認 | `GET /slots` |
| 保存 | `POST /slots/0?action=save` body: `{"filename": "{hash}.bin"}` |
| 復元 | `POST /slots/0?action=restore` body: `{"filename": "{hash}.bin"}` |
| 消去 | `POST /slots/0?action=erase` |

### 5.3 制約・既知の限界

* **モデル依存**: Snapshot はモデル・ctx_size・量子化が一致しないと再利用不可
* **マルチモーダル**: vision 対応モデルでは slot save が未対応の場合がある（v0.3 はテキストモデルのみサポート）
* **ディスク容量**: Snapshot 1 件あたり数百 MB〜数 GB（ctx 長に比例）
* **llama-server 再起動**: ディスク restore により RAM 上の KV を復元可能（`--slot-save-path` 必須）

## 7. データ形式

### 7.1 Snapshot meta.yaml

```yaml
version: 1
prompt_hash: sha256:abc123...       # Prefix 本文の SHA-256
created_at: "2026-06-28T12:00:00+09:00"
build_name: Prompt                  # build.yaml name
model:
  path: models/your-model.gguf      # llama.yaml の model
  fingerprint: sha256:def456...     # GGUF ファイル hash（任意・推奨）
llama:
  ctx_size: 4096
  server_url: http://127.0.0.1:8080
kv:
  slot_id: 0
  filename: abc123....bin
  prefix_tokens: 2847               # save 時の usage から記録
metrics:
  snapshot_ttft_ms: 3200.5          # Snapshot 作成時の prefill 時間
  snapshot_total_ms: 3500.0
```

### 7.2 Test Suite（regression/suites/*.yaml）

```yaml
version: 1
name: 基本人格チェック
description: 挨拶・自己紹介・禁止事項の確認

# snapshot: latest | explicit hash | auto
snapshot: latest

defaults:
  temperature: 0.7
  max_tokens: 256

cases:
  - id: greeting
    input: こんにちは
    matchers:
      - type: not_contains
        value: AIアシスタント
      - type: regex
        pattern: "(こんにちは|はじめまして)"

  - id: no_meta
    input: あなたはAIですか？
    matchers:
      - type: not_contains
        value: ChatGPT
      - type: not_contains
        value: 言語モデル

  - id: intro_length
    input: 自己紹介して
    matchers:
      - type: min_length
        value: 20
      - type: max_length
        value: 500

  - id: golden
    input: 好きな食べ物は？
    expected_file: ../fixtures/food_answer.txt
    match_mode: normalize_whitespace   # exact | normalize_whitespace | substring
```

### 7.3 Matcher 種別（v0.3）

| type | パラメータ | 説明 |
|------|------------|------|
| `contains` | `value` | 部分一致 |
| `not_contains` | `value` | 含まない |
| `regex` | `pattern`, `flags?` | 正規表現 |
| `exact` | `value` | 完全一致 |
| `min_length` | `value` | 文字数下限 |
| `max_length` | `value` | 文字数上限 |
| `expected_file` | `path`, `match_mode` | fixtures との比較 |

複数 matcher は **すべて Pass** でケース Pass（AND）。

### 7.4 Run report.json

```yaml
version: 1
run_id: "20260628T120530Z-abc"
suite: default.yaml
snapshot:
  prompt_hash: sha256:abc123...
  stale: false                      # 実行時点で Prefix が変わっていたか
started_at: "..."
finished_at: "..."
summary:
  total: 5
  passed: 4
  failed: 1
  skipped: 0
cases:
  - id: greeting
    status: pass
    input: こんにちは
    output: "..."
    matchers: [...]
    usage:
      prompt_tokens: 12
      completion_tokens: 45
      ttft_ms: 85.2
      total_ms: 1200.0
  - id: no_meta
    status: fail
    failures:
      - matcher: not_contains
        value: 言語モデル
        message: "output contains forbidden phrase"
```

## 8. Backend API

プレフィックス: `/api/regression`

### 8.1 Snapshot

| Method | Path | 説明 |
|--------|------|------|
| `GET` | `/snapshot/status` | 現在 Prefix に対する Snapshot の有無・鮮度 |
| `POST` | `/snapshot` | 現在 Prefix から Snapshot 作成（KV save 含む） |
| `DELETE` | `/snapshot/{prompt_hash}` | Snapshot 削除 |

**GET /snapshot/status レスポンス例**

```json
{
  "prefix_tokens_estimate": 2847,
  "current": {
    "prompt_hash": "sha256:abc123",
    "has_snapshot": true,
    "snapshot_created_at": "2026-06-28T12:00:00+09:00",
    "kv_file_exists": true
  },
  "latest_run": {
    "run_id": "...",
    "passed": 4,
    "failed": 1,
    "finished_at": "..."
  }
}
```

### 8.2 Suite

| Method | Path | 説明 |
|--------|------|------|
| `GET` | `/suites` | スイート一覧 |
| `GET` | `/suites/{name}` | スイート詳細 |
| `PUT` | `/suites/{name}` | スイート保存 |
| `DELETE` | `/suites/{name}` | スイート削除 |

### 8.3 Run

| Method | Path | 説明 |
|--------|------|------|
| `POST` | `/run` | スイート実行 |
| `GET` | `/runs` | 実行履歴（直近 N 件） |
| `GET` | `/runs/{run_id}` | レポート詳細 |

**POST /run リクエスト**

```json
{
  "suite": "default.yaml",
  "snapshot": "latest",
  "options": {
    "stop_on_first_failure": false,
    "ensure_snapshot": true
  }
}
```

`ensure_snapshot: true` のとき、Snapshot が無い or stale なら自動で Snapshot 更新を試行する。

### 8.4 LLM リクエスト形式（内部）

Snapshot prefill / ケース実行共通:

```json
{
  "model": "gpt-3.5-turbo",
  "messages": [
    { "role": "system", "content": "<Prefix>" },
    { "role": "user", "content": "<Test Input>" }
  ],
  "id_slot": 0,
  "cache_prompt": true,
  "stream": false,
  "temperature": 0.7,
  "max_tokens": 256
}
```

v0.2 の RUN TEST（Prefix 全文を user に入れる）とは異なる。Regression では **system + user 分離** を標準とする。

### 9.2 操作

| UI | 動作 |
|----|------|
| **Update Snapshot** | `POST /api/regression/snapshot` |

## 10. 鮮度（Stale）判定

Snapshot を **stale** とみなす条件（いずれか）:

1. `prompt_hash` が現在 Prefix と不一致
2. `config/llama.yaml` の model / ctx_size が meta と不一致
3. KV ファイル（`.bin`）が存在しない
4. llama-server が `--slot-save-path` 未設定で restore 不可

stale 時の UI:

* 警告バナー: 「プロンプトが変更されています。Snapshot を更新してください」
* Run 時: `ensure_snapshot: true` なら自動更新、否则 WARN 付きで実行

## 11. エラーハンドリング

| 状況 | HTTP | ユーザー向けメッセージ |
|------|------|------------------------|
| LLM 未設定 | 503 | LLM 設定を行ってください |
| llama-server 未起動 | 503 | `npm run llama` で起動してください |
| slot save 未対応 | 502 | `--slot-save-path` を llama.yaml に追加してください |
| Snapshot 作成失敗 | 502 | KV 保存に失敗しました（ログ参照） |
| ケース timeout | 504 | ケース `{id}` がタイムアウトしました |
| matcher 設定不正 | 422 | スイート YAML の検証エラー |

Run 全体は **ケース単位で失敗を許容** する。1 ケース失敗でも残りを実行（`stop_on_first_failure: true` 時を除く）。

## 12. 非機能要件

| 項目 | 目標 |
|------|------|
| Snapshot 更新 | Prefix 3000 tok で TTFT 計測・表示（環境依存のため SLA なし） |
| ケース実行（KV 再利用時） | Prefix prefill をスキップし TTFT を Snapshot 比 90% 以上削減（目安） |
| スイート 10 ケース | 直列実行、進捗を UI にストリーム表示（v0.3.1 で SSE 検討） |
| 同時 Run | v0.3 では 1 実行のみ（Backend ロック） |

## 15. セキュリティ・運用

* Snapshot / KV ファイルにユーザーデータは含めない（Prefix + モデル状態のみ）
* `workspace/regression/` はローカル開発用。リモート共有しない
* Suite YAML は Git 管理し、人格テストの意図をコードレビュー可能にする
* `.bin` ファイルは `.gitignore` 必須（容量・モデル依存）

---

## 16. 受け入れ基準（Phase 1）

1. Components 編集 → Prefix 変更後、Snapshot 更新で新 KV が保存される
2. 同一 Prefix で 5 ケース実行時、2 ケース目以降の `prompt_tokens`（Suffix 分のみ増加）が Snapshot 単体実行より大幅に少ない
3. `not_contains` matcher により意図的な人格崩れを Fail 検出できる
4. llama-server 再起動後も、保存済み Snapshot から Run できる
5. LLM 未設定時は Regression UI が設定を促し、Backend が 503 を返す
6. 既存 RUN TEST（Single Test）の動作に回帰がない
